## 作为模组管理器

### 管理器和加载器

（外观）模组使用 XXMI（3dmigoto） 模组加载器加载，模组加载器会将 mods 文件夹中的所有模组加载到游戏中。并且为了支持部分模组的持久化功能，模组加载器会在另外一个地方创建一个记录配置的文件，用来记录一些模组的配置项。

但是加载器有一些已知的问题：
1. 即使两个mod共用了大多数资源、但是模组之间并没有共享机制，每一个模组都需要单独保存一份模型文件，这导致非常占空间。
2. 无法实现动态管理，比如模组的启用和禁用需要通过移动文件来实现，这导致非常麻烦。如果能够通过链接文件来实现动态管理，那么就可以通过简单的启用和禁用来管理模组，而不需要移动文件。
3. 所有的mod必须放在同一个磁盘中，这导致不能将mod放在不同的磁盘中（如果可以放在某些网盘中，那就更好了），这导致非常麻烦。
4. 每次都需要手动更新mod，而无法通过源下载位置自动订阅模型更新。

### 模组管理器功能

1. 配置驱动的行为，用户可以通过配置文件来配置不同游戏的模组安装位置，以及模组的启用和禁用等行为。比如用户可以配置某个游戏的模组安装位置为 D:\Games\GameName\Mods，另一个游戏的模组安装位置为 E:\Games\GameName\Mods。
2. 可以配置多个模组安装位置，用户可以将模组安装在不同的磁盘中，甚至可以将模组安装在网盘中。
3. 智能压缩：对于一些体积较大的模组，用户可以选择将其压缩成zip文件来节省空间，模组管理器会自动解压缩并加载这些模组。我们的归档是基于 数据chunk 的存储，也就是说，当多个文件拥有一致的数据块（不用是完整的文件，只要是文件的一部分），我们就只存储一份数据块，然后其他文件就通过引用的方式来使用这个数据块，这样就可以大大节省空间。系统自动根据分配的空间和模组的使用频率智能压缩，从而在用户体验（因为每次启动/切换模组都需要解压缩）和存储空间之间取得平衡。
4. 模组更新：用户可以通过模组管理器来订阅模组的更新，当模组有更新时，模组管理器会提示用户更新，并且可以自动下载和安装更新。
5. 可视化的数据维护：用户可以通过模组管理器来查看和维护模组的数据，比如配置模组的昵称、tag、category、描述、封面等信息，以及查看模组的使用情况，比如启用的模组、禁用的模组、最近使用的模组等。
6. （皮肤）模组一般会有自己的快捷键用于切换不同的模式。我们可以尝试自动识别、记录和管理这些快捷键，以便用户更方便地使用模组。同时支持用户通过覆写的方式来修改这些快捷键，以适应不同用户的使用习惯。
7. 模组分类：用户可以通过模组管理器来对模组进行分类，比如按照类型、主题、作者等进行分类，以便更方便地管理和查找模组。
8. 冲突检测：当用户启用多个模组时，模组管理器可以检测这些模组之间是否存在冲突，比如是否对同一hash的素材进行了修改，如果存在冲突，模组管理器可以提示用户，并且提供解决冲突的方案，比如选择启用哪个模组或者通过覆盖配置的方式来解决冲突。
9. preset >= pack > mod：用户可以通过模组管理器来创建和管理preset、pack和mod，preset是一个预设，包含了一组模组的配置，用户可以通过切换preset来快速切换不同的模组配置，pack是一个模组包，包含了一组模组，用户可以将一组mod打包成一个pack来方便管理和分享，mod是一个模组，包含了一个或多个文件，用户可以通过模组管理器来创建和管理这些mod。其中 preset可以引用其他preset、pack和mod，pack可以引用mod，mod只能包含文件。通过这种层级关系，用户可以更方便地管理和分享模组。比如 角色A 的 天使装扮mod + 翅膀mod + 十字架武器mod 可以打包成一个pack，命名为 角色A的天使装扮pack，然后用户可以创建一个 preset，命名为 天使系列 preset，引用 角色A的天使装扮pack 和 角色B的天使装扮pack，这样用户就可以通过切换 天使系列 preset 来快速切换角色A和角色B的天使装扮了。在选择启用时，preset 只能够单选，而 pack 和 mod 可以多选，这样就可以通过 preset 来快速切换不同的模组配置了。
10. 商店兼容：用户可以在我们这里登入其他网站的账号（比如 nexus），来同步和管理在其他网站上下载的模组（类似 ehviewer 一样）通过代理和爬取商店的数据，然后在我们这使用我们的页面风格提供浏览和下载等功能，并且可以将下载的模组直接添加到我们的模组管理器中来进行管理，并且可以通过模组管理器来订阅这些模组的更新，当模组有更新时，模组管理器会提示用户更新，并且可以自动下载和安装更新。
11. 模组分享：用户可以通过模组管理器来分享自己的模组配置，比如将一个 preset 或者 pack 分享给其他用户，其他用户可以通过模组管理器来导入这个 preset 或者 pack 来快速使用这个模组配置了。

### 美学设计

模组管理器使用了 流行的设计风格，并且使用了大胆的配色和灵活的动效来提升用户体验。美学设计是我们的特色之一，我们希望通过美学设计来吸引用户，并且让用户在使用模组管理器的过程中感受到愉悦和满足。我们会不断优化和改进我们的美学设计，以提供更好的用户体验。

## QA

### 1. TS + Rust 的分工是怎样的？

核心逻辑全部用 Rust 实现。

为什么推荐全 Rust：

chunk 去重存储需要高效二进制处理，Rust 比 Node.js 更适合
Windows 符号链接在 Rust 中更可控
CLI 和 GUI 完美复用：CLI 和 Tauri 都直接调用同一个 core crate
Tauri 本身就是 Rust 后端，架构天然契合

> 但是我对于 rust 不是很熟悉，所以说你可能还是得考虑一下，如果你认为你拥有十足的信心能够在不用我的检查的情况下独立完成 rust 的开发，那么我觉得全 rust 是一个不错的选择了，如果你觉得你可能会在 rust 的开发过程中遇到一些问题，那么我觉得还是应该考虑一下使用 ts 来实现逻辑和功能，因为我熟悉 ts 的开发，并且我也可以在 ts 的开发过程中提供一些帮助和支持，我们的核心性能可能只是在解压缩的时候会有一些性能要求，其他的功能可能并不需要特别高的性能，大部分都是简单的逻辑处理。

### 2. MVP 应该包含哪些功能？

1. **智能压缩（第3点）**：基于数据 chunk 的去重存储是个很好的想法，但实现复杂度很高。你打算：
   - 自己实现这套 chunk 存储系统？
   - 还是基于现有方案（如 btrfs/zfs 的去重、或类似 restic 的 content-addressable storage）？
  
> 我认为应该使用第一性原则来看，谁能够更好地实现这个功能，那就用什么方案，到时候我会给你一些实际使用中的mod，你可以自行分析这些mod的文件结构和数据块分布来设计这个chunk存储系统。至于选什么技术方案，我觉得不应该预设立场，最重要的是能够实现这个功能，并且性能和用户体验都能够满足要求。

1. **商店兼容（第10点）**：爬取 Nexus 等网站可能涉及法律风险和反爬机制。你是否考虑过：
   - 使用官方 API（Nexus 有 API，但有限制）？
   - 这个功能的优先级如何？

> 我们相当于是提供一个模组管理器的浏览器，用户可以通过这个浏览器来访问 Nexus 等网站，并且在这个浏览器中提供下载和管理模组的功能，这样就不涉及爬取和反爬机制了。或者提供一个浏览器插件，用户可以点击这个插件提供的按钮来将当前页面的模组添加到我们的模组管理器中来进行管理，这样也不涉及爬取和反爬机制了。至于优先级，我觉得这个功能虽然很有用，但可能不是 MVP 的核心功能，我们可以在后续迭代中添加这个功能。

1. **preset/pack/mod 层级（第9点）**：这个设计很清晰，但我想确认：
   - pack 内的 mod 是否可以单独禁用？还是 pack 是原子单位？
   - preset 之间是否允许嵌套引用？（可能导致循环依赖）

> pack 内的 mod 可以单独禁用，pack 只是一个模组包，包含了一组 mod，用户可以将一组 mod 打包成一个 pack 来方便管理和分享，但是用户也可以选择启用 pack 中的某些 mod，而禁用其他 mod，这样就可以更灵活地管理模组了。preset 之间允许嵌套引用，但是需要注意循环依赖的问题，而且问题比你想得简单的多，因为 即使是 A、B 来回引用，我们也只是取 A 和 B 的 mod 的并集来启用而已，所以并不会导致什么问题了。

然后再迭代添加智能压缩、商店兼容等高级功能。
