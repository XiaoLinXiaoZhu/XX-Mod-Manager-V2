# XX Mod Manager - 产品愿景与功能路线图

> 本文档记录了项目的完整功能需求、技术决策和优先级规划。
> 当前实现进度请参考 [progress.md](./progress.md)。

## 背景：加载器的已知问题

（外观）模组使用 XXMI（3dmigoto） 模组加载器加载，模组加载器会将 mods 文件夹中的所有模组加载到游戏中。并且为了支持部分模组的持久化功能，模组加载器会在另外一个地方创建一个记录配置的文件，用来记录一些模组的配置项。

但是加载器有一些已知的问题：
1. 即使两个mod共用了大多数资源、但是模组之间并没有共享机制，每一个模组都需要单独保存一份模型文件，这导致非常占空间。
2. 无法实现动态管理，比如模组的启用和禁用需要通过移动文件来实现，这导致非常麻烦。如果能够通过链接文件来实现动态管理，那么就可以通过简单的启用和禁用来管理模组，而不需要移动文件。
3. 所有的mod必须放在同一个磁盘中，这导致不能将mod放在不同的磁盘中（如果可以放在某些网盘中，那就更好了），这导致非常麻烦。
4. 每次都需要手动更新mod，而无法通过源下载位置自动订阅模型更新。

## 功能需求清单

### 1. 配置驱动的行为 ✅ 已规划
用户可以通过配置文件来配置不同游戏的模组安装位置，以及模组的启用和禁用等行为。比如用户可以配置某个游戏的模组安装位置为 D:\Games\GameName\Mods，另一个游戏的模组安装位置为 E:\Games\GameName\Mods。

### 2. 多位置存储 ✅ 已规划
可以配置多个模组安装位置，用户可以将模组安装在不同的磁盘中，甚至可以将模组安装在网盘中。

### 3. 智能压缩（Chunk 去重存储） ✅ 已实现
对于一些体积较大的模组，用户可以选择将其压缩成zip文件来节省空间，模组管理器会自动解压缩并加载这些模组。我们的归档是基于 数据chunk 的存储，也就是说，当多个文件拥有一致的数据块（不用是完整的文件，只要是文件的一部分），我们就只存储一份数据块，然后其他文件就通过引用的方式来使用这个数据块，这样就可以大大节省空间。系统自动根据分配的空间和模组的使用频率智能压缩，从而在用户体验（因为每次启动/切换模组都需要解压缩）和存储空间之间取得平衡。

### 4. 模组更新订阅 📋 待开发
用户可以通过模组管理器来订阅模组的更新，当模组有更新时，模组管理器会提示用户更新，并且可以自动下载和安装更新。

### 5. 可视化数据维护 📋 待开发
用户可以通过模组管理器来查看和维护模组的数据，比如配置模组的昵称、tag、category、描述、封面等信息，以及查看模组的使用情况，比如启用的模组、禁用的模组、最近使用的模组等。

### 6. 快捷键管理 ✅ 已实现
（皮肤）模组一般会有自己的快捷键用于切换不同的模式。我们可以尝试自动识别、记录和管理这些快捷键，以便用户更方便地使用模组。同时支持用户通过覆写的方式来修改这些快捷键，以适应不同用户的使用习惯。

### 7. 模组分类 📋 待开发
用户可以通过模组管理器来对模组进行分类，比如按照类型、主题、作者等进行分类，以便更方便地管理和查找模组。

### 8. 冲突检测 ✅ 已实现
当用户启用多个模组时，模组管理器可以检测这些模组之间是否存在冲突，比如是否对同一hash的素材进行了修改，如果存在冲突，模组管理器可以提示用户，并且提供解决冲突的方案，比如选择启用哪个模组或者通过覆盖配置的方式来解决冲突。

### 9. Preset/Pack/Mod 层级 ✅ 已实现
preset >= pack > mod：
- **mod**: 包含一个或多个文件的模组
- **pack**: 一组 mod 的集合（pack 内 mod 可单独禁用）
- **preset**: 预设配置，可引用 mod、pack 和其他 preset（单选切换）

例如：角色A 的 天使装扮mod + 翅膀mod + 十字架武器mod 可以打包成一个pack，命名为 角色A的天使装扮pack，然后用户可以创建一个 preset，命名为 天使系列 preset，引用 角色A的天使装扮pack 和 角色B的天使装扮pack。

preset 之间允许嵌套引用，取并集处理，不会导致循环依赖问题。

### 10. 商店兼容 📋 待开发（后续迭代）
用户可以在我们这里登入其他网站的账号（比如 nexus），来同步和管理在其他网站上下载的模组。实现方案：
- 方案A：提供一个内嵌浏览器，用户通过浏览器访问 Nexus 等网站并下载管理模组
- 方案B：提供浏览器插件，用户点击按钮将当前页面的模组添加到管理器中

### 11. 模组分享 📋 待开发（后续迭代）
用户可以通过模组管理器来分享自己的模组配置，比如将一个 preset 或者 pack 分享给其他用户，其他用户可以通过导入来快速使用这个模组配置。

## 美学设计

模组管理器使用了流行的设计风格，并且使用了大胆的配色和灵活的动效来提升用户体验。美学设计是我们的特色之一，我们希望通过美学设计来吸引用户，并且让用户在使用模组管理器的过程中感受到愉悦和满足。

## 技术决策 QA

### TS + Rust 的分工

核心逻辑全部用 Rust 实现。推荐理由：
- chunk 去重存储需要高效二进制处理，Rust 比 Node.js 更适合
- Windows 符号链接在 Rust 中更可控
- CLI 和 GUI 完美复用：CLI 和 Tauri 都直接调用同一个 core crate
- Tauri 本身就是 Rust 后端，架构天然契合

> **实际决策**：核心性能敏感部分（chunk 去重存储）使用 Rust 实现，其余逻辑功能使用 TypeScript 实现，便于快速迭代和团队协作。

### 前端框架选择：React + Tauri

**决策**：前端使用 React（TSX）+ Tauri，从零构建 UI，不复用 legacy 分支的 Vue 代码。

**理由**：
- legacy 分支的前端代码设计需要大幅重构，从零开始更干净
- React 生态下的 AI 辅助工具链更成熟：[Agentation](https://www.agentation.com/) 可直接定位 TSX 文件生成 prompt，react-scan 可实时分析渲染性能
- 开发者作为审查者只需通过工具看到性能问题和 UI 反馈即可，不需要深入每行 React 代码
- TSX 文件的 AI 可读性和可编辑性优于 SFC（.vue 文件）

**UI 设计文档**：详见 [docs/ui/design.md](../ui/design.md)
